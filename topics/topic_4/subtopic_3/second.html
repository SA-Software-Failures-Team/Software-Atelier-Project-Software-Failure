The Technical Failure: Excel's Silent Data Loss

Many labs submitted their data via a legacy process that funneled results into a single .XLSX file. While the modern XLSX format can hold over a million individual rows, PHE's specific template, for reasons related to its legacy data structure, had an effective limit of only 65,536 rows per sheet - a limit inherited from the older and previously used .XLS format.


When this file reached the maximum capacity, the system did not crash or sound a loud alarm; instead, it failed silently. Any data rows beyond the 65,536th were simply removed and permanently lost, vanishing from the official dashboard and, most critically, from the queue for the contact tracing team that used this information to contain the spread of the virus.


For days, these 15,841 infected individuals were unaware of their status, and the system, designed to isolate the infected and trace their closest contacts, was operating on incomplete and missing information. The silent failure meant that operators had no indication that data was being lost, allowing the problem to persist undetected.

This core technical failure was compounded by the usage of Microsoft Excel as a database. Excel is well known as a powerful tool for analysis and visualization, but it is not designed to be a robust, transactional database management system (DBMS) designed for this type of large scale, automated data recording. Any proper DBMS would have rejected additional data once a table was full, triggering an immediate error alarming the user about the issue.
